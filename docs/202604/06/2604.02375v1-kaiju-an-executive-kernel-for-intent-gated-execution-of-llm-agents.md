## 摘要
基于 ReAct 的大语言模型（LLM）工具调用自主智能体存在三个局限性：串行延迟、上下文二次方增长，以及易受提示词注入和幻觉的影响。近期研究趋向于将规划与执行分离，但在各种情况下，模型仍与执行机制耦合。我们为 LLM 智能体引入了一种系统级抽象，将智能体工作流的执行与 LLM 推理层解耦。我们定义了两个一等抽象：(1) 意图门控执行（IGX），一种在执行时强制执行意图的安全范式；(2) 执行内核（Executive Kernel），负责管理调度、工具分发、依赖解析、故障处理和安全。在 KAIJU 中，LLM 预先进行规划，利用依赖感知的参数注入并行地进行乐观工具调度。工具通过 IGX 进行授权，授权基于四个独立变量：范围、意图、影响和许可（外部批准）。KAIJU 支持三种自适应执行模式（Reflect、nReflect 和 Orchestrator），提供适用于复杂调查、深度分析或研究的渐进式细粒度执行控制。针对 ReAct 基准的实证评估表明，由于规划开销，KAIJU 在简单查询上存在延迟惩罚，在适度复杂度下趋于收敛，而在需要并行数据采集的计算查询上具有结构性优势。除了延迟之外，这种分离还强制执行了 ReAct 仅通过提示词无法实现的行为保证。代码开源于 https://github.com/compdeep/kaiju

## Abstract
Tool-calling autonomous agents based on large language models using ReAct exhibit three limitations: serial latency, quadratic context growth, and vulnerability to prompt injection and hallucination. Recent work moves towards separating planning from execution but in each case the model remains coupled to the execution mechanics. We introduce a system-level abstraction for LLM agents which decouples the execution of agent workflows from the LLM reasoning layer. We define two first-class abstractions: (1) Intent-Gated Execution (IGX), a security paradigm that enforces intent at execution, and (2) an Executive Kernel that manages scheduling, tool dispatch, dependency resolution, failures and security. In KAIJU, the LLM plans upfront, optimistically scheduling tools in parallel with dependency-aware parameter injection. Tools are authorised via IGX based on four independent variables: scope, intent, impact, and clearance (external approval). KAIJU supports three adaptive execution modes (Reflect, nReflect, and Orchestrator), providing progressively finer-grained execution control apt for complex investigation and deep analysis or research. Empirical evaluation against a ReAct baseline shows that KAIJU has a latency penalty on simple queries due to planning overhead, convergence at moderate complexity, and a structural advantage on computational queries requiring parallel data gathering. Beyond latency, the separation enforces behavioural guarantees that ReAct cannot match through prompting alone. Code available at https://github.com/compdeep/kaiju

### 1. 核心问题与研究动机
*   **核心问题**：传统的 ReAct（推理-行动）架构在处理复杂任务时存在三大瓶颈：
    1.  **串行延迟**：工具调用通常是顺序的，导致总耗时随步骤线性增长。
    2.  **上下文爆炸**：每轮对话都会携带历史记录，导致上下文以二次方速度 $O(n^2k)$ 增长，既昂贵又易导致模型注意力分散。
    3.  **安全性脆弱**：工具调用的安全性依赖于提示词（Prompt），模型容易受到提示注入攻击或产生幻觉，且模型能观察到安全拦截结果，从而进行针对性绕过。
*   **研究背景**：现有的并行执行方案（如 LLM Compiler）缺乏中间自适应能力和结构化安全门控。

### 2. 方法论：核心思想与技术细节
KAIJU 引入了**执行内核（Executive Kernel）**和**意图门控执行（IGX）**两个核心抽象：
*   **解耦架构**：将 LLM 视为无状态资源。推理层负责规划，执行层负责依赖解析、工具分发、故障恢复和安全强制执行。
*   **执行内核（Executive Kernel）**：
    *   **图执行**：将任务分解为有向无环图（DAG）。
    *   **参数注入（Param Refs）**：允许在规划阶段定义数据流（如“步骤2使用步骤1的输出”），在执行阶段自动注入具体值，无需 LLM 介入。
    *   **自适应模式**：提供 **Reflect**（波次间反射）、**nReflect**（每 N 个节点反射）和 **Orchestrator**（逐节点观察）三种模式，平衡速度与精度。
*   **意图门控执行（IGX）**：基于四个独立变量进行安全校验：
    1.  **范围（Scope）**：允许调用的工具白名单。
    2.  **意图（Intent）**：调用者的权限等级（由系统而非 LLM 设定）。
    3.  **影响（Impact）**：工具操作的风险等级（只读、写入、删除）。
    4.  **许可（Clearance）**：委托给外部 HTTP 端点的资源级授权。
*   **结构化隔离**：LLM 无法看到安全门控的决策过程，阻断了攻击者通过模型反馈探测安全边界的路径。

### 6. 主要结论与发现
*   **性能优势**：在复杂和计算型任务中，KAIJU 的 `nReflect` 模式显著快于 ReAct（例如复杂任务 9.5s vs 28.9s）。
*   **上下文优化**：Token 复杂度从 $O(n^2k)$ 降低到 $O(nk)$，有效避免了长任务中的上下文耗尽问题。
*   **可靠性提升**：在天文学基准中，KAIJU 完成率为 100%，而 ReAct 为 80%（因上下文溢出失败）。
*   **安全性**：IGX 机制在不增加显著延迟（亚毫秒级）的情况下，实现了确定性的安全保障。

### 7. 优点
*   **结构化安全**：通过将安全逻辑移出 LLM 上下文，从根本上解决了提示注入导致的策略绕过问题。
*   **高效并行**：利用依赖感知注入，实现了比传统 ReAct 更高效的工具并行化。
*   **行为确定性**：通过执行内核强制执行“不达目的不罢休”的重试逻辑，减少了模型因遇到困难而过早放弃的现象。

### 8. 不足与局限
*   **简单任务开销**：对于只需 1-2 步的简单查询，KAIJU 的规划开销使其延迟高于 ReAct。
*   **规划依赖性**：系统高度依赖 Planner（规划器）的分解能力，如果初始规划极差，会增加反射轮数和延迟。
*   **动态影响分类风险**：对于通用工具（如 Shell），其风险评估依赖模式匹配，可能存在被巧妙构造的命令绕过的风险。
*   **应用限制**：目前主要针对工具密集型任务，对于纯逻辑推理任务的提升可能有限。
