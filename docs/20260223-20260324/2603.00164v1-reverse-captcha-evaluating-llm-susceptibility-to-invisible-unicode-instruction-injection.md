## 摘要
我们引入了 Reverse CAPTCHA，这是一个评估框架，用于测试大语言模型是否会遵循嵌入在看似正常文本中的不可见 Unicode 编码指令。与区分人类和机器的传统验证码（CAPTCHA）不同，我们的基准测试利用了一种能力差距：模型能够感知人类读者不可见的 Unicode 控制字符。我们评估了来自两个供应商的五个模型，涵盖了两种编码方案（零宽二进制和 Unicode 标签）、四个提示级别、两种有效载荷框架，以及启用或禁用工具调用的情况。在 8,308 个模型输出中，我们发现工具调用显著增强了合规性（Cohen's h 高达 1.37，具有显著效应），模型表现出特定于供应商的编码偏好（OpenAI 模型解码零宽二进制；Anthropic 模型更倾向于 Unicode 标签），并且显式解码指令在单一模型和编码下可将合规性提高多点达 95 个百分点。所有两两模型间的差异均具有统计学显著性（p < 0.05，经过 Bonferroni 校正）。这些结果凸显了一个尚未被充分探索的攻击面，即通过不可见 Unicode 有效载荷进行的提示词注入。

## Abstract
We introduce Reverse CAPTCHA, an evaluation framework that tests whether large language models follow invisible Unicode-encoded instructions embedded in otherwise normal-looking text. Unlike traditional CAPTCHAs that distinguish humans from machines, our benchmark exploits a capability gap: models can perceive Unicode control characters that are invisible to human readers. We evaluate five models from two providers across two encoding schemes (zero-width binary and Unicode Tags), four hint levels, two payload framings, and with tool use enabled or disabled. Across 8,308 model outputs, we find that tool use dramatically amplifies compliance (Cohen's h up to 1.37, a large effect), that models exhibit provider-specific encoding preferences (OpenAI models decode zero-width binary; Anthropic models prefer Unicode Tags), and that explicit decoding instructions increase compliance by up to 95 percentage points within a single model and encoding. All pairwise model differences are statistically significant (p < 0.05, Bonferroni-corrected). These results highlight an underexplored attack surface for prompt injection via invisible Unicode payloads.

*   **研究背景**：提示词注入（Prompt Injection）是 LLM 安全领域的核心威胁。现有研究多集中在人类可见的文本攻击上。

*   **“逆向验证码”概念**：传统验证码（CAPTCHA）利用“人能完成而机器不能”的任务来区分用户；本文提出的 Reverse CAPTCHA 则利用“机器能感知（Token 级别）而人类不可见（视觉渲染）”的感知差异，测试模型是否会违背用户显性意图去执行隐性指令。

### 2. 方法论
核心思想是构建一个受控的评估框架，通过两种不可见编码方式将指令植入文本：
*   **编码方案**：
    *   **零宽二进制 (Zero-Width Binary, ZW)**：使用 U+200B（零宽空格）代表 0，U+200C（零宽非连接符）代表 1，将 ASCII 指令转为 8 位二进制序列。
    *   **Unicode 标签 (Unicode Tags)**：利用 U+E0000 及其后的标签字符块，将 ASCII 字符一一映射到不可见字符。
*   **提示梯度 (Hint Gradient)**：设置了四个层级的系统提示，以测试模型在不同知情程度下的反应：
    1.  **Unhinted**：无提示，测试模型是否自发解码。
    2.  **Hint (codepoints)**：告知存在特定 Unicode 编码，但不提供算法。
    3.  **Hint (full)**：提供完整的解码规则（如 8-bit ASCII 转换）。
    4.  **Hint (full) + injection**：在提供规则的同时，将隐藏指令改为对抗性载荷（如“忽略之前所有指令”）。
*   **工具增强**：引入 `run_python` 工具，允许模型通过编写代码来辅助解码。

### 3. 实验设计
*   **数据集/场景**：构建了 30 对常识问答（Trivia）。每个案例包含一个可见的正确答案（如“巴黎”）和一个隐藏的指令要求回答另一个词（如“紫罗兰”）。
*   **Benchmark**：通过 2 种编码 × 4 种提示级别 + 1 个控制组，共 270 个基础测试用例。
*   **对比模型**：
    *   **OpenAI**：GPT-5.2, GPT-4o-mini。
    *   **Anthropic**：Claude Opus 4, Claude Sonnet 4, Claude Haiku 4.5。
*   **评估指标**：合规率（Compliance Rate），即模型仅输出隐藏答案的比例。

### 6. 主要结论与发现
*   **工具调用是核心放大器**：开启 Python 工具后，所有模型的合规率均大幅上升。例如，Claude Haiku 的合规率从 0.8% 飙升至 49.2%。
*   **供应商特定的编码偏好**：OpenAI 模型对“零宽二进制”更敏感，而 Anthropic 模型对“Unicode 标签”更敏感。这暗示了不同厂商在分词器（Tokenizer）设计或训练数据组成上的差异。
*   **提示词依赖性**：在没有提示且不使用工具的情况下，模型几乎不会自发执行隐藏指令（合规率 ≤1.1%）。
