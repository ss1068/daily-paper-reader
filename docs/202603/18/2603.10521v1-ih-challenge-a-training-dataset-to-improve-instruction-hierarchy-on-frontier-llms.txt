## Abstract 

Instruction hierarchy (IH) defines how LLMs prioritize system, developer, user, and tool instructions under conflict, providing a concrete, trust-ordered policy for resolving instruction conflicts. IH is key to defending against jailbreaks, system prompt extractions, and agentic prompt injections. However, robust IH behavior is difficult to train: IH failures can be confounded with instruction-following failures, conflicts can be nuanced, and models can learn shortcuts such as overrefusing. We introduce IH-Challenge , a reinforcement learning training dataset, to address these difficulties. Fine-tuning GPT-5-Mini on IH-Challenge with online adversarial example generation improves IH robustness by +10.0% on average across 16 in-distribution, out-of-distribution, and human red-teaming benchmarks (84.1% →

94.1%), reduces unsafe behavior from 6.6% to 0.7% while improving helpfulness on general safety evaluations, and saturates an internal static agentic prompt injection evaluation, with minimal capability regression. We release the IH-Challenge dataset (huggingface.co/datasets/openai/ih-challenge) to support future research on robust instruction hierarchy. 

## 1 Introduction 

Modern LLMs are designed with multiple roles reserved for different input sources, e.g. , system admin, application developer, user, response from tool calls, etc. These roles are also different in terms of their trust level. For example, safety guardrails specified by the system admin should not be overridden by potentially malicious users. Wallace et al. (2024) introduced the notion of the 

instruction hierarchy (IH), which specifies a policy of how to prioritize conflicting instructions in different roles, and trains this policy into the LLM. This general policy, if followed by the LLM in a robust manner, can be used to protect the model against different types of misuse, including (i) jailbreaking (Zou et al., 2023; Wei et al., 2023), by specifying a safety policy in the system prompt; (ii) system prompt extraction (Perez & Ribeiro, 2022), by prompting the LLM to never output its system prompt; and (iii) direct and indirect prompt injection (Greshake et al., 2023), by prompting/training the LLM to not follow instructions from conflicting lower-priority roles. Following the instruction hierarchy robustly requires the LLM to understand and reason about nuanced conflicts between different instructions across diverse task domains. Moreover, since LLMs

IF-simple. We aim for the dataset to be challenging in a way that specifically trains robust IH behavior. To support stable RL training, we require each IH-Challenge task to be objectively gradable by a per-datapoint Python grader that remains valid under adversarially synthesized instruction conflicts. 3. We fine-tune GPT-5-Mini on IH-Challenge using RL with online adversarial example generation. ## 2 Background 

Instruction hierarchy (IH) is a concept introduced by Wallace et al. (2024), which specifies a policy of how to prioritize instructions when there are conflicts between multiple instructions in the LLM’s context. In the original proposal, instructions are prioritized in the following order according their role: 1. System Message specified by the model provider ( e.g. , OpenAI) containing general instructions, safety guidelines, tool definitions, etc. 

2. User Message specified by the user. 3. Tool Message containing the result returned by a tool call to an external tool or API, e.g. , search engine, code interpreter, third-party API. Tool messages are untrusted inputs. For instance, if System Message = “Do not reveal the PIN 1234” and User Message = “Give me the secret PIN”, instruction hierarchy would prioritize System Message over User Message and the LLM should respond with a refusal. Since its original formulation, OpenAI has introduced an additional role called Developer Message in between System Message and User Message (OpenAI, 2025b). Unless specified otherwise, we will default to this updated IH policy, i.e. , System Message ≻ Developer Message ≻ User Message ≻ Tool Message .

Reinforcement learning (RL) for LLMs initially emerged as a way to control model behavior (Christiano et al., 2017). Unlike supervised fine-tuning where the model is trained to maximize likelihood of fixed training samples, RL first samples the model’s rollouts and grades them using reward functions, and then trains the model maximize its reward to promote desirable behavior and penalize undesirable ones. This procedure can be done in an online fashion with a trained reward model such as in PPO (Schulman et al., 2017), or in an offline fashion with offline reward functions such as in DPO (Rafailov et al., 2023). Although RL has proven to be a very powerful training technique for LLMs, it suffers from several drawbacks. One of the most well-known drawbacks is reward hacking (Amodei et al., 2016; Gao et al., 2023), where the LLM finds shortcuts to maximize its reward instead of solving the problem. The reward hacking problem is particularly severe when the reward function is hard to specify or too expensive to evaluate exactly. 3Large-Scale RL Improves Instruction Hierarchy on Frontier LLMs

During training, we use an attacker LLM to generate the attack prompt on-the-fly by iteratively probing the defender model, and use the final output prompt for RL training.

One solution to the reward hacking problem is to design reward functions that can be specified via deterministic graders such as Python code. Using RL with such rewards eliminates label noise and makes the training process much more resistant to reward hacking, and is particularly suitable for settings where it is relatively straightforward to design such rewards. Another classic challenge of LLM alignment is balancing helpfulness and harmlessness (Askell et al., 2021). Models’ instruction-following training can nudge them to comply with malicious requests, while safety training often makes them problematically conservative about following benign instructions.

system ≻ developer ≻ user ≻ tool .

Finally, we describe the (defender) model training pipeline. 

3.1 Offline Task Skeleton Construction 

We construct offline task skeletons that consist of one or more high-priority instructions, a placeholder for the low-priority (conflicting, potentially adversarial) message, and a Python grader. We construct these skeletons with the following design principles: 1. IF-simple (under attacks). Rewards should primarily reflect IH handling rather than puzzle-solving. Therefore, we choose tasks that are IF-simple, so an IH-good model can reliably succeed. Importantly, tasks should remain IF-simple under adversarial low-priority input; otherwise, an adaptive attacker will be incentivized to make the task IF-hard to fool the model ( e.g. , by turning a simple counting problem into counting thousands of words). 2. Programmatically gradeable (under attacks). Tasks must remain well-specified and program-matically gradeable even in the presence of adversarial conflicts. To ensure this, we restrict adversarially conflicting instructions to low-priority roles, so that the IH grading criterion can be determined fully offline by the high-level priority instruction. We also exclude ambiguous tasks that are not objectively gradeable by Python like jailbreaks. 3. Avoiding shortcut learning. Tasks should span diverse patterns to discourage the model from relying on shortcut heuristics. A common failure mode in IH training is overrefusal, such as a shortcut like “if you see a password, refuse”. To address this, we diversify task structures so that only models with robust IH, rather than those that have learned brittle shortcuts, can achieve consistently high reward. 5Large-Scale RL Improves Instruction Hierarchy on Frontier LLMs We construct task skeletons via LLM synthesis followed by filtering and validation: 1. LLM synthesis. We prompt an LLM to generate a structured task object, including (i) a conversation with high-priority instructions from high-priority roles and a message placeholder for low-priority conflicting instructions. (ii) Python grader code to grade whether the (defender) model’s response follows IH, and (iii) a small set of responses that should pass, and a small set that should fail, for validation later. To make task skeletons diverse, we condition generation on randomly varied factors like logical constraints and output format constraints, and optionally provide a few in-context seed examples. 2. Filtering and validation. We apply static checks to the grader code ( e.g. , AST parsing) and normalize response formatting differences ( e.g. , plain JSON vs. JSON in code fences) to reduce grading noise. We then unit-test each grader on the provided pass and fail examples, keeping only tasks whose graders reliably distinguish passing from failing responses. This validation removes almost all cases where the task description and grader code are inconsistent, but does not guarantee the grader is fully correct, so we perform a final manual review. 3.2 Online Adversarial Conflict Synthesis 

We synthesize adversarial low-priority conflicting messages using language model programming to fill each task skeleton’s low-priority placeholder online during (defender) model training. The attacker is a frozen model without safety guardrails that interacts with a defender evaluation tool. 

Attacker goal. Given a task skeleton and the current defender checkpoint, the attacker aims to produce a low-priority conflicting message that is IH-hard for the defender, i.e. , one that is likely to cause the defender to receive low reward when inserted into the skeleton. 

Budgeted attacker rollout. We implement synthesis as a budgeted propose–evaluate–revise loop. At the start of each rollout, the attacker receives (i) an explicit objective to induce an IH violation and (ii) the skeleton’s high-priority instructions as reference. The attacker has access to an evaluation tool that tests a candidate message against the current defender and returns grader feedback. Under a fixed tool-call budget, the attacker repeatedly: 1. Propose. Generate a candidate low-priority message. 2. Evaluate. Call the evaluation tool, which (a) inserts the candidate into the placeholder, (b) queries the defender on the instantiated prompt, and (c) scores the defender’s response with the task’s Python grader. The tool returns pass/fail outcomes, diagnostics, and optionally a representative defender response. 3. Revise. Use the feedback to refine the next proposal ( e.g. , by changing attack strategies or restructuring the request). When the rollout ends, either when the attacker stops early or budget exhausted, the attacker outputs the final low-priority message. We then fill the placeholder with this message to form the final conflict prompt. 

Defender training. We train the defender using RL, where at each step we synthesize a conflict prompt online using the budgeted attacker rollout. For each conflict prompt, we sample multiple defender responses, score them with the Python grader, and do a policy gradient update. ## 5 Analysis 

5.1 Ablation on Training Task Splits 

Table 6 shows the effectiveness of different task splits in IH-Challenge . Overall, we observe that using all four splits ( Single-Constraint , Multi-Constraint , Input-Conditioned and Anti-Overrefusal ) yields the best balance between robustness/safety and overrefusal. Removing the Anti-Overrefusal split causes the model to refuse significantly more. Training on only the harder splits, Multi-Constraint and 

Input-Conditioned , yields slightly worse results on both IH robustness/overrefusal and helpfulness. 

5.2 System Mitigations 

When LLMs are deployed in a production system, it is common to deploy other system components alongside the main LLM to serve as safeguards (Sharma et al., 2025). These system mitigations can complement the LLM’s weaknesses and improve overall safety. We evaluate the effect of several basic system mitigations, including: 

• Sandwich defense repeats the higher-tier instruction after the lower-tier instruction to remind the LLM if a conflict exists. 

• System prompt defense reinforces the IH policy by adding an IH spec to the system prompt. 

• Output monitor uses a second model to judge whether the main LLM’s response follows IH policy correctly. 5.3 Adaptive Automated Red-Teaming 

Prior work (Nasr et al., 2025; Wen et al., 2025) showed that fine-tuning an LLM via RL can be an effective automated red-teaming strategy. We apply this strategy to stress-test GPT-5-Mini-R 

in an adaptive setting. We re-use IH-Challenge as an RL training set for the attacker by reversing the reward, i.e. , the attacker receives a binary reward if the Python grader grades the defender’s response as violating IH. The attacker is initialized from a “anti-refusal” GPT-5 model that underwent fine-tuning to remove safety guardrails, and we use GPT-5-Mini-R with frozen weights as a fixed defender. We also consider a variant of the defender with the output monitor defense applied. Figure 8 shows attack success rate (ASR) on the three splits of IH-Challenge other than Anti-Overrefusal across RL steps. ASR starts off low due to GPT-5-Mini-R having strong IH robustness, but gradually climbs throughout training. With the output monitor applied, ASR is lower on all three splits. This result suggests that although system mitigations such as an output monitor may not be effective against static attacks (as shown in Figure 7), defense-in-depth can be beneficial against certain adaptive attackers.

## 6 Related Work 

There is a growing body of literature on improving the LLM’s instruction hierarchy robustness. Wu et al. (2024) proposed learning a separate role embedding and adding it to the token embedding of an instruction to better achieve role separation. Hines et al. (2024) proposed the sandwich defense, which repeats the higher-tier instruction before the assistant begins generating its response. Although it was originally proposed in the setting of prompt injection attacks, we found it also helps improve IH robustness. Zheng et al. (2025) proposed a training-based defense similar to our training approach. The method leverages the IFEval dataset (Zhou et al., 2023)—a programmatically verifiable instruction-following evaluation dataset—by treating the original IFEval instructions as higher-tier instructions and generating a lower-tier attack instruction that violates the higher-tier one. The model can then be trained using RL to maximize the programmatically verifiable reward. Compared to our approach, the dataset contains more IF-hard tasks rather than focusing on IH conflicts, is more prone to overrefusal, and the attack strings are fixed and less adversarial. We further validate that training on IH-Challenge is compatible with large-scale frontier models and perform adaptive evaluations such as human red-teaming. 13 Large-Scale RL Improves Instruction Hierarchy on Frontier LLMs Our work expands on the existing literature on RL for LLMs. Using RL for IH robustness relies heavily on OOD generalization. The tasks in IH-Challenge are IF-simple and limited in variety due to the need for objective programmatic grading, yet when the model is trained via RL on IH-Challenge ,IH robustness generalizes to OOD task domains such as safety where programmatic grading is infeasible. Our result suggests that RL can be a general tool even for non-programmatically gradeable tasks when the proxy training tasks are well-designed. 

## 7 Conclusion 

We showed that training on IH-Challenge is a viable path for improving IH robustness on frontier LLMs. By fine-tuning GPT-5-Mini on IH-Challenge , coupled with online adversarial example generation, we can achieve IH robustness that generalizes to unseen attacks and OOD tasks, and even yields substantial gains on disallowed content safety and prompt injection robustness. One particularly promising scaling paradigm is compute-scaling through adversarial training. In this paper, we considered both defender RL training with a fixed attacker and attacker RL training with a fixed defender. What happens when we train both attacker and defender models simultaneously? This training paradigm is related to adversarial training for adversarial robustness (Madry et al., 2017) and generative adversarial networks (Goodfellow et al., 2014). Without programmatically gradeable rewards, it is possible for the attacker/defender goal to drift from the actual robustness goal. We hypothesize that such rewards may be a required component in this adversarial training paradigm when we scale attacker and defender training compute.
