## Abstract 

Suffix jailbreak attacks serve as a systematic method for red-teaming Large Language Models (LLMs) but suffer from prohibitive computational costs, as a large number of candidate suffixes need to be evaluated before identifying a jailbreak suf-fix. This paper presents Prefix-Shared KV Cache (PSKV), a plug-and-play inference optimization technique tailored for jailbreak suffix generation. Our method is motivated by a key observation that when performing suffix jailbreaking, while a large number of candidate prompts need to be evaluated, they share the same targeted harmful instruction as the prefix. Therefore, instead of performing re-dundant inference on the duplicated prefix, PSKV maintains a single KV cache for this prefix and shares it with every candidate prompt, enabling the parallel inference of diverse suffixes with min-imal memory overhead. This design enables more aggressive batching strategies that would other-wise be limited by memory constraints. Exten-sive experiments on six widely used suffix attacks across five widely deployed LLMs demonstrate that PSKV reduces inference time by 40% and peak memory usage by 50%, while maintaining the original Attack Success Rate (ASR). The code has been submitted and will be released publicly.

Our method is motivated by a key struc-tural redundancy in jailbreak optimization: while a batch of candidate suffixes varies, they typically share an identi-cal targeted harmful instruction as the prefix. Standard KV cache implementations ignore this property, redundantly duplicating the long prefix cache for every candidate. In contrast, PSKV maintains a single, shared KV cache for the invariant prefix and broadcasts it to all candidate suffixes during inference. By employing a suffix-centric alignment strategy, PSKV supports dynamic expansion and parallel batched loss computation. Crucially, unlike encapsulated inference engines, PSKV integrates seamlessly into existing attack pipelines, ensuring substantial acceleration and mini-mal memory overhead while preserving gradient access. ## 2. Related Work 

Automated Suffix Jailbreaking. Recent research has tran-sitioned from manual red-teaming to automated adversarial optimization. The foundational method, GCG (Zou et al., 2023), employs a greedy, gradient-guided search to itera-tively refine an adversarial suffix. Building on this, subse-quent works have explored more sophisticated search strate-gies. BEAST (Sadasivan et al., 2024) replaces the greedy algorithm with beam search to explore a wider range of promising candidates. Zhu’s AutoDAN (Zhu et al., 2023) utilizes genetic algorithms to evolve suffixes, mitigating the local optima problem of purely greedy methods. Other variants like GCQ (Gradient-based Attack with Quantiza-tion) (Hayase et al., 2024) focus on improving the efficiency of the core gradient-based update step itself. An alternative paradigm uses LLMs themselves as gener-ators. AdvPrompter (Paulus et al., 2025) fine-tunes a lan-guage model to automatically compose adversarial prompts. Similarly, AmpleGCG trains a generative model to directly output effective suffixes, amortizing the search cost into a one-time training process (Liao & Sun, 2024). Although these methods differ, they all require evaluating a large num-ber of candidate suffixes to create a training set during either the model training or data generation phases. Our work presents an orthogonal optimization that accelerates this critical evaluation step, benefiting both categories of attack.

## 4. Methodology 

In this section, we first illustrate the redundant computation problem of prefix KV vectors in suffix jailbreaking and then introduce our PSKV method to tackle the raised challenge and substantially improve the efficiency of suffix jailbreak attacks. 

4.1. Redundant Prefix computation in Suffix Jailbreak Attacks 

According to the general jailbreak suffix synthesis process described in Section 3, the main computational burden of suffix jailbreaking arises from performing K · q inference passes (in the response generation phase) in each synthesis iteration, where K is the number of candidate suffixes in each iteration, and q is the number of new suffixes generated for each candidate. Larger K and q usually lead jailbreak attacks to explore more candidate suffixes, thus increasing the chance of finding an effective jailbreak suffix; however, they also result in more LLM inference times. The most straightforward implementation, which we term the unaccelerated traditional attack method , treats each of the K ·q candidate prompts as independent inputs in every iteration. This approach repeatedly re-computes the KV vectors for the prefix x(h), leading to significant redundant computational overhead. As a result, when an LLM is performing inference on these prompts in parallel, it will redundantly calculate the same KV vectors for the same prefix x(h) for up to K · q times at each step. A more advanced baseline would employ a standard KV cache implementation. The KV vectors for the prefix x(h)

are computed only once before the iterative synthesis pro-cess begins. However, to evaluate all K · q suffix candidates in a single parallel batch, this pre-computed prefix cache must be duplicated K · q times to match the batch dimension of the suffixes, which introduces a severe memory bottle-neck. For a batch of B prompts, this problem is exacerbated, requiring B · K · q copies of the caches. Furthermore, as shown in Table 1, the average token length of instructions in HarmBench is 78.4 (Mazeika et al., 2024), which is significantly larger than the standard jailbreak suffix length in practice (typically set to 20 tokens). Collectively, these observations suggest that if the KV vectors of the prefix harmful instruction x(h) can be calculated only once and used for the inference of different candidate prompts, then both the computation and memory costs of suffix jail-breaking can be considerably reduced, as a large amount of redundant intermediate KV vectors no longer need to be calculated. Fortunately, we observe that while the number of candidate prompts that an LLM needs to perform inference on in each jailbreaking iteration is as large as K · q, these prompts all share the same prefix, i.e. , the original harmful instruction 

x(h). Specifically, as illustrated in Fig. 1, the suffix jailbreak attack only modifies the suffixes of these candidate prompts but keeps the harmful instruction x(h) as their prefix intact.

4.2. PSKV for Single Jailbreak Suffixes Synthesis 

These observations highlight a clear opportunity for opti-mization. Based on our previous analysis, we now propose 

PSKV , a method that can substantially accelerate existing suffix jailbreak attacks by reducing redundant computation of prefix KV vectors during the candidate prompt inference process of suffix jailbreaking. The overall procedures are also illustrated in Figure 2. PSKV adopts a lazy, layer-wise expansion strategy. Un-like standard implementations that pre-allocate expanded memory for all layers during initialization, we maintain a compact, single-batch storage of the prefix. Specifically, be-fore a suffix jailbreak attack starts the iterative suffix search-ing process in Figure. 1, PSKV runs a single forward pass for the harmful instruction x(h) to compute and cache its KV vectors across all attention layers of the targeted LLM. Then, when the targeted LLM is performing inference on the 

K · q candidate prompts during the iterative suffix searching process, whenever the LLM needs to use the KV vectors of the prefix from an attention layer, PSKV directly retrieves the corresponding KV vectors from its cache and duplicates these vectors K · q times for the LLM to support its infer-ence process. Compared with vanilla suffix jailbreaking, our PSKV method reduces the time-consuming prefix KV cache computation from K · q times to only a single time ,which can significantly improve the computation efficiency of suffix jailbreaking. Furthermore, we note that the KV vector duplicating pro-cedure in PSKV requires little memory overhead. This is because during LLM inference, the outputs of all LLM attention layers are calculated in a layer-by-layer manner, and the computation process only requires the prefix KV vectors from that specific layer. As a result, the prefix KV vector duplication procedure in PSKV is also performed in a layer-by-layer manner, and at any given time, only KV vectors from a single layer are duplicated. This ensures that at any given moment, the memory-intensive expansion is isolated to a single layer, considerably reducing peak memory overhead compared to the vanilla approach where redundant prefix repetitions persist across the entire network stack simultaneously.

5.2. Results 

In this section, we study the effectiveness of PSKV in accel-erating suffix jailbreak attacks from different aspects. 

PSKV does not affect the attack performance. According to the design of PSKV described in Section 4, PSKV is a performance-neutral acceleration method that does not affect the attack performance of accelerated suffix jailbreak attacks. This is because PSKV only removes redundant key-value vectors and does not change the algorithm itself or the computational accuracy. Nevertheless, we conduct experiments here to empirically justify this. We collect and present the ASRs of the jailbreak attacks that without any accelerations and those that adopt PSKV in Table 3. From the table, we observe minor fluctuations in ASR be-tween the baseline and PSKV-accelerated attacks. These variations stem not from PSKV altering the core attack logic but from the inherent stochasticity (e.g, heuristic search algo-rithms or sampling-based generation of multiple responses for ASR computation) in the end-to-end evaluation pipeline. Overall, the attack performance remains broadly consistent across all cases, demonstrating that PSKV effectively ac-celerates the attacks without systematically harming their original effectiveness. 

PSKV considerably accelerates suffix jailbreaking. Ta-ble 4 details the time consumption of varying accelera-tion strategies.First, for optimization-based attacks that per-form only one or two forward passes per candidate suffix step (including GCG, GCQ, BEAST, and AmpleGCG), our PSKV (PSKV) consistently achieves a speed-up ratio of ap-proximately 1.4× to 1.6× compared to the no-acceleration baseline.Notably, this acceleration becomes even more pro-nounced for Zhu’s AutoDAN, where the speed-up ratio rises to a range of ∼ 1.8× to 1.95 × (e.g., reducing time from 88 to 45 minutes on Llama-3-8B). This performance gain is attributed to the mechanism of KV cache acceleration: it reduces computational redundancy by storing states. Since Zhu’s AutoDAN requires substantially more inference steps per iteration (due to its hierarchical genetic algorithm), there is a larger proportion of redundant prefix computation for PSKV to eliminate. Conversely, for AdvPrompter, the speed-up ratio decreases to approximately 1.3×. This is because AdvPrompter in-volves synchronous reinforcement learning; the total time includes both inference and generator training. While PSKV accelerates the inference phase, the training over-head remains constant, diluting the overall proportion of time saved. Furthermore, compared to the Standard KV Cache, our method maintains a comparable speed without incurring time penalties (in some cases, such as Llama-3 Ad-vPrompter, it is even faster). This indicates that the overhead caused by our temporary KV tensor expansion is negligible relative to the total inference time. ## C. Computational Efficiency Analysis of AdvPrompter 

C.1. Observations 

In our experimental evaluation of AdvPrompter (Table 4), we observed two notable phenomena: 1. Severe performance degradation without KV caching: All models exhibit substantially higher latency in the “None” (no acceleration) setting compared to cached configurations. This section analyzes the algorithmic and architectural factors underlying these observations. 12 Accelerating Suffix Jailbreak attacks with Prefix-Shared KV-cache 

C.2. The Cost of Cache-less Decoding 

The severe slowdown in the “None” setting stems from the fundamental inefficiency of autoregressive generation without KV caching. 

Quadratic Total Complexity. With KV caching, generating token t requires computing attention only between the new query and cached keys/values, yielding O(1) complexity per layer with respect to history length. Without caching, the model must recompute all previous hidden states, requiring O(t) operations at step t. Summing over a sequence of length n,the total complexity degrades from O(n) to O(n2).

Amplification by AdvPrompter’s Search Strategy. Unlike gradient-based attacks ( e.g. , GCG) that perform single forward-backward passes, AdvPrompter employs beam search with sampling during suffix generation. This expands the effective batch size multiplicatively (batch × beam × samples), causing the quadratic recomputation penalty to be incurred across many parallel sequences simultaneously. The resulting computational burden saturates GPU throughput and explains the disproportionate slowdown observed in the “None” setting. 

C.3. Architectural Factors in Llama-Series Latency 

While all models benefit from KV caching, Llama-2 and Llama-3 exhibit consistently higher latency than Mistral and Qwen. We attribute this to two architecture-specific factors. 

Llama-2: Multi-Head Attention Overhead. Llama-2-7B employs Multi-Head Attention (MHA) with 32 key-value heads, whereas Mistral-7B and Qwen2.5-7B utilize Grouped-Query Attention (GQA) with 8 key-value heads. In cache-less decoding, the model must recompute K and V projections for the entire sequence history at each generation step. With 4×

more KV heads, Llama-2 incurs proportionally higher memory bandwidth requirements for loading projection weights and storing intermediate KV tensors. This memory-bound overhead is particularly pronounced when batch sizes are large, as in AdvPrompter’s beam search. 

Llama-3: Vocabulary Projection Overhead. Llama-3-8B adopts GQA, mitigating the MHA-related overhead. However, it employs a vocabulary of 128K tokens—approximately 4× larger than Llama-2 and Mistral ( ∼32K). AdvPrompter uniquely requires training a generator model, which involves computing full logit distributions over the vocabulary and backpropagating through the output projection layer. The computational cost of this Rd → R|V | projection and its gradient scales linearly with vocabulary size, imposing a persistent overhead on Llama-3 that cannot be alleviated by KV caching alone. The ”None” baseline forces redundant recomputation across the entire sequence history at each decoding step, a cost amplified by AdvPrompter’s beam search strategy. Llama-2 is further penalized by its MHA architecture, which demands higher memory bandwidth than GQA-based models. Llama-3, while adopting GQA, suffers from the computational burden of its large vocabulary during AdvPrompter’s generator training phase. Our PSKV effectively eliminates redundant prefix computations, providing consistent speedups across all evaluated models.
